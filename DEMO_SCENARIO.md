## ЧАСТЬ 1 — Введение (1–2 минуты)

**[Экран: README.md проекта в VS Code]**

> «Добрый день. Меня зовут Максим, и я покажу проект Legal-RAG — систему интеллектуального поиска по российскому законодательству на основе технологии Retrieval-Augmented Generation.
>
> Проект решает конкретную бизнес-задачу: юристы и обычные граждане тратят огромное количество времени на поиск нужных статей в кодексах. Legal-RAG позволяет задать вопрос на естественном языке и получить точный ответ с указанием конкретных статей и ссылок на источники.
>
> Сейчас в системе загружены 5 нормативных актов: Гражданский кодекс, Семейный кодекс, Жилищный кодекс, Трудовой кодекс и Закон о защите прав потребителей — в сумме более 3000 текстовых фрагментов.»

## ЧАСТЬ 2 — Архитектура (4–5 минут)

**[Экран: docker-compose.microservices.yml или схема в draw.io/Mermaid]**

> «Система построена на микросервисной архитектуре из 6 независимых компонентов. Покажу каждый.»

### 2.1. API Gateway (порт 8080)

**[Экран: `scripts/start_microservices.py` → показать оркестратор]**

> «Центральный узел — API Gateway на FastAPI. Он оркестрирует запуск всех сервисов в правильном порядке:
> сначала Storage и Cache → затем Inference и Search → и последним поднимается сам Gateway.
> Реализованы JWT-аутентификация, rate limiting (100 запросов в минуту), CORS, health-мониторинг всех сервисов.»

**[Открыть браузер: http://localhost:8080/docs — показать Swagger]**

> «Вот Swagger-документация Gateway — здесь видны все эндпоинты: `/api/query` для вопросов, `/api/upload` для загрузки документов, `/admin/*` для управления, `/health/all` для мониторинга.»

### 2.2. Embeddings Service (порт 8001)

**[Экран: `services/embeddings/embeddings_server.py`]**

> «Сервис векторизации использует модель sbert_large_nlu_ru от Сбера — это специализированная модель для русского языка c размерностью 1024. Она превращает текст в числовые векторы, что позволяет сравнивать смысловую близость документов. API совместим с форматом OpenAI, поэтому легко интегрируется.»

### 2.3. Три базы данных

**[Экран: docker-compose.yml — показать секции postgres, chromadb, neo4j]**

> «Используется три базы данных, каждая для своей задачи:
>
> — **PostgreSQL** хранит метаданные документов, текст чанков, пользователей, логи запросов.
>
> — **ChromaDB** — векторное хранилище. Здесь лежат 3033 эмбеддинга для семантического поиска по смыслу.
>
> — **Neo4j** — графовая база. В ней хранятся связи между статьями: "статья 21 ГК ССЫЛАЕТСЯ на статью 18 ГК", "определяет понятие дееспособности". Это позволяет находить связанные нормы, которые текстовый поиск бы пропустил.»

### 2.4. RAG-пайплайн — ключевая логика

**[Экран: `core/langgraph_rag_workflow.py` — показать граф состояний]**

> «А вот самое интересное — RAG-пайплайн, построенный на LangGraph. Это граф состояний, где каждый узел выполняет свою задачу:»

**[Показать схему потока — можно нарисовать заранее или показать код]**

> «1. **Анализ запроса** — классификация: это вопрос про определение, процедуру или сравнение? Извлечение ключевых юридических терминов.
>
> 2. **Гибридный поиск** — одновременно работают два механизма:
>    — BM25 (точное совпадение терминов) — вес 60%
>    — Семантический поиск (по смыслу через векторы) — вес 40%.
>    Это критично для юридических текстов, где один термин может означать разное в разных кодексах.
>
> 3. **Графовое расширение** — Neo4j добавляет связанные статьи, на которые ссылаются найденные нормы.
>
> 4. **Оценка качества** — система проверяет, достаточно ли контекста для ответа. Если нет — включается механизм CRAG: переформулирует запрос и ищет повторно.
>
> 5. **Генерация ответа** — Google Gemini 2.5 Flash формирует ответ в юридическом формате IRAC (Issue → Rule → Application → Conclusion).
>
> 6. **Верификация фактов** — LegalFactVerifier проверяет, что все утверждения в ответе подтверждены источниками. Если находит противоречие — помечает это.
>
> 7. **Оценка уверенности** — confidence score от 0 до 100%.»

### 2.5. Почему именно такой подход?

> «Почему не просто "запихнуть всё в ChatGPT"?
>
> — Во-первых, **галлюцинации**. LLM может выдумать несуществующую статью. Поэтому RAG + верификация — обязательны.
>
> — Во-вторых, **точность терминологии**. Чистый семантический поиск путает "исковую давность" и "срок давности по уголовным делам". BM25 ловит точные совпадения.
>
> — В-третьих, **связи между нормами**. Статья 21 ГК ссылается на статью 27 — граф это знает, обычный поиск — нет.
>
> Альтернативы, которые рассматривал: RAG на простом LangChain (слишком линейно, нет self-correction), чистый full-text search (теряет смысл), только семантический поиск (теряет точную терминологию). Гибридный подход с LangGraph дал лучший результат — 97.5% точности на бенчмарке из 40 вопросов.»

## ЧАСТЬ 3 — Живая демонстрация (5–7 минут)

### 3.1. Админ-панель

**[Открыть: http://localhost:8090]**

> «Начнём с админ-панели на Streamlit. Здесь видны все загруженные документы — 5 кодексов, 3033 чанка. Можно загрузить новый документ, удалить существующий, запустить переиндексацию.
>
> Панель поддерживает PDF, TXT, DOCX. При загрузке документ автоматически разбивается на чанки по 800-1200 символов с учётом структуры русского текста, извлекаются метаданные — номер статьи, название закона — и всё индексируется в ChromaDB.»

**[Показать список документов, можно кликнуть на один]**

### 3.2. Чат-интерфейс — простой вопрос

**[Открыть: http://localhost:8501]**

> «Теперь главное — чат. Зададим вопрос...»

**[Ввести: "Что такое исковая давность и каков общий срок?"]**

> «Видите — система показывает шаги работы в реальном времени: анализ запроса, поиск, генерация, верификация.
>
> Ответ: исковая давность — это срок для защиты права по иску лица, право которого нарушено. Общий срок — 3 года. Указаны статьи 195 и 196 ГК РФ. Оценка уверенности — около 70%.
>
> Обратите внимание: каждое утверждение подкреплено конкретной статьёй. Система не "помнит" — она цитирует.»

### 3.3. Вопрос посложнее — межкодексовый

**[Ввести: "Что является совместной собственностью супругов?"]**

> «Этот вопрос интереснее — он требует информации из двух кодексов: Семейного (статья 34) и Гражданского (статья 256). Давайте посмотрим, найдёт ли система оба источника...
>
> Да — ответ ссылается на оба кодекса. Это заслуга гибридного поиска и графового расширения: Neo4j знает, что статья 34 СК связана со статьёй 256 ГК.»

### 3.4. Вопрос из другого кодекса

**[Ввести: "Какие права у потребителя при обнаружении недостатков товара?"]**

> «Вопрос по Закону о защите прав потребителей. Ответ ссылается на статью 18 — право на замену, ремонт, возврат денег. Уверенность — около 78%.
>
> Важно: система не просто нашла статью — она структурировала ответ: сначала правовая норма, потом варианты действий, потом ограничения. Это формат IRAC.»

### 3.5. Вопрос, на который нет ответа

**[Ввести: "Какова продолжительность ежегодного отпуска?"]**

> «А вот намеренно сложный случай. В нашей базе только выдержка из Трудового кодекса — часть III про трудовой договор. Статья 115 про отпуск туда не вошла.
>
> Система честно показывает низкую уверенность и предупреждает, что информация может быть неполной. Это важно — система не врёт и не выдумывает, когда данных нет.»

## ЧАСТЬ 4 — Результат для бизнеса (2–3 минуты)

**[Экран: вернуться в VS Code или показать слайд с метриками]**

> «Какую задачу решает Legal-RAG?
>
> **Задача:** юрист тратит 15-30 минут на поиск нужной статьи в кодексе, перечитывая десятки страниц. С Legal-RAG — это 30 секунд на вопрос и точный ответ с цитатами.
>
> **Метрики проекта:**
>
> — **97.5% точность** на бенчмарке из 40 вопросов (верифицировано вручную)
> — **8 из 10** тестовых вопросов отвечены корректно с указанием статей (2 оставшихся — из-за неполноты загруженного Трудового кодекса, а не ошибки системы)
> — **Среднее время ответа ~30 секунд** — включая поиск, генерацию и верификацию
> — **67% cache hit rate** — повторные похожие запросы отвечаются мгновенно
>
> **Автоматизация:**
>
> — Загрузка нового кодекса занимает 2 минуты через админ-панель — система автоматически разбивает текст, создаёт эмбеддинги, индексирует
> — Масштабирование: добавить новый нормативный акт не требует изменений кода
> — Верификация фактов исключает галлюцинации — критично для юридической сферы
>
> **Потенциальный эффект:** если в юридическом отделе 10 юристов делают по 20 поисков в день — экономия ~50 человеко-часов в месяц. Плюс снижение риска пропустить связанную норму.»
